AI AGENT IMPLEMENTATION BRIEF
Project: Random Video Chat App with Google Sign-In
Version: MVP v1
Author intent: Build the app exactly around the idea described below, while keeping the architecture clean enough to scale later.

======================================================================
1. PRODUCT IDEA IN PLAIN WORDS
======================================================================

Build a mobile-first random video chat app where people sign up with Google, open the app, tap "Start Chat", and then see other users one by one as candidate profiles.

The intended experience is this:

- Assume 10 users are online.
- Each user opens the app and clicks Start Chat.
- The app begins showing each user the profiles of other currently-available users one by one.
- For every shown profile, the user can either Accept or Reject.
- If one user accepts but the other user does not accept yet, no call starts yet.
- If both users accept each other, then a video call starts immediately.
- During the call, if one user swipes left, the call ends.
- After the call ends, that user returns to the waiting/matching flow and starts seeing the next profiles again.
- The system should continue doing this in a loop as long as the user stays available.

This is not a public room app. This is not a one-to-many livestream app.
This is a 1-to-1 random matching app with mutual acceptance before the call.

======================================================================
2. IMPORTANT CONCEPTS THE IMPLEMENTATION MUST UNDERSTAND
======================================================================

The original idea mentions:
- webhooks
- WebRTC
- WebRTC hooks

These are not the same thing. The implementation must treat them correctly.

A) WebRTC
WebRTC is the actual browser/device technology that carries the audio/video stream between users. It handles media, peer connection setup, ICE candidates, NAT traversal, tracks, and connection state.

B) Signaling
WebRTC still needs a signaling mechanism to exchange:
- SDP offer
- SDP answer
- ICE candidates
- call state messages

Signaling is not built into WebRTC. We must provide it ourselves using our backend or a realtime service.

C) Webhooks
Webhooks are server-to-server event notifications. They are useful for:
- room started
- room ended
- participant joined
- participant left
- analytics
- cleanup
- moderation logging
- async business logic

Webhooks are NOT the primary way to do instant user-to-user accept/reject interaction and are NOT the thing that carries audio/video.

D) React hooks / WebRTC hooks
These are frontend abstractions to make the UI easier to build. Examples:
- useRoomState
- useParticipant
- useTracks
- custom hooks like useMatchingQueue, useLocalMedia, useCallSession

Hooks improve developer experience. They are not the media layer themselves.

Conclusion:
- Use WebRTC (directly or through a managed product like LiveKit) for audio/video.
- Use realtime signaling for negotiation and call state.
- Use webhooks only for backend lifecycle handling, analytics, cleanup, moderation, and auditing.
- Use hooks on the frontend to manage UI state cleanly.

3. RECOMMENDED ARCHITECTURE FOR MVP
======================================================================

Use this architecture unless there is a very strong reason not to:

Frontend:
- Next.js or React web app
- Mobile-first responsive UI
- TypeScript
- Tailwind CSS
- Zustand or React Context for local app state

Backend / platform:
- Supabase Auth for Google sign-in
- Supabase Postgres for database
- Supabase Realtime for online presence, queue state, and app-level signaling/events
- Supabase Storage for profile avatars if needed
- Supabase Edge Functions for protected server logic
- LiveKit Cloud for video rooms and media transport
- LiveKit webhooks for room lifecycle updates

Why this is the best MVP choice:
- Google sign-in is straightforward through Supabase.
- Supabase gives auth, DB, realtime, storage, and server logic in one stack.
- LiveKit reduces the complexity of raw WebRTC room management.
- For a 1-to-1 app, LiveKit gives production-grade video faster than building raw peer logic from scratch.
- The frontend can still use hooks and clean state.
- The system can be launched faster and with fewer fragile networking issues.

Alternative path:
If you absolutely want raw WebRTC instead of LiveKit, implement:
- Supabase Realtime Broadcast as signaling
- Supabase Presence for online/ready state
- STUN/TURN servers
- custom RTCPeerConnection lifecycle
This is possible, but more error-prone and slower for an AI coding agent to get right.

Therefore:
DEFAULT IMPLEMENTATION PATH = Supabase + LiveKit

6. USER FLOW IN DETAIL
======================================================================

A) Sign in
1. User opens app.
2. User clicks "Continue with Google".
3. Supabase Auth handles Google OAuth.
4. On first login, create a profile row for the user.
5. User lands on the home screen.

B) Home screen
User sees:
- their profile summary
- online indicator
- Start Chat button
- optional settings
- report/block history access
- camera/microphone permission status

C) Start Chat
1. When user clicks Start Chat:
   - ask for camera/mic permissions if needed
   - mark user as queueing / available
   - publish realtime presence
   - start candidate feed logic
2. The system begins showing candidates one by one.

D) Candidate card
The candidate card can show:
- photo
- display name
- age
- country or city if enabled
- bio or interest tags if enabled
- online now indicator

Actions:
- Accept
- Reject
- Block
- Report

E) Accept flow
1. User A sees User B and taps Accept.
2. Save A -> B decision as accept.
3. Check whether B has already accepted A within the active matching window.
4. If yes:
   - create a mutual match
   - reserve both users atomically
   - create call session
   - generate video room/token
   - push realtime "match_found" event to both users
5. If no:
   - keep A in waiting_for_mutual_accept or continue browsing depending on UX policy

Recommended UX policy:
- After A accepts B, A may continue browsing other candidates unless the system chooses a more exclusive model.
- However, once a mutual match is formed, all other pending candidate actions for both users must be invalidated safely.

F) Reject flow
1. Save A -> B decision as reject.
2. Do not show B again for a cooldown period (for example 24 hours or longer depending on product rules).
3. Immediately move to the next candidate.

G) Mutual accept
If A accepted B and B accepted A:
- lock both users
- create the session
- stop showing more candidates
- transition both to matched_pending_room and then in_call

H) During call
Show:
- local video
- remote video
- mute button
- camera toggle
- speaker toggle
- end call button
- left-swipe gesture to end
- report/block shortcut

I) End call
If either side swipes left or presses end:
- end session for both
- leave the room
- persist call end reason
- mark users back to online_idle or queueing based on UX
- by default, return them to queue if they still want to continue chatting

======================================================================
7. MATCHING LOGIC REQUIREMENTS
======================================================================

The candidate engine must not be random chaos. It needs rules.

Base matching rules for MVP:
- only show users who are online and queueing
- do not show self
- do not show blocked users
- do not show users already rejected during cooldown
- do not show users already in active call
- do not show users already exhausted recently unless recycling is explicitly allowed
- prefer users not seen recently
- prefer users with compatible availability right now
- ensure one active reservation per user

Recommended candidate order:
1. online users in queue
2. not previously rejected
3. not previously shown very recently
4. not blocked
5. not currently reserved
6. optional randomization for fairness

Important concurrency rule:
Two users may be accepted by multiple people around the same time.
The backend must atomically reserve users when forming a mutual match.

This means:
- use a transaction or server-side guarded function
- before creating a match, re-check both users are still free
- if either is already reserved or in call, abort and continue searching

8. DATABASE DESIGN
======================================================================

Use Supabase Postgres.

Suggested tables:

1) profiles
Purpose:
Store public user profile data.

Columns:
- id uuid primary key references auth.users(id)
- username text unique nullable
- full_name text nullable
- avatar_url text nullable
- bio text nullable
- age int nullable
- country text nullable
- is_profile_complete boolean default false
- created_at timestamptz default now()
- updated_at timestamptz default now()

2) user_presence
Purpose:
Current app-level presence snapshot.

Columns:
- user_id uuid primary key references profiles(id)
- status text not null
  allowed values: offline, online_idle, queueing, browsing_candidates, waiting_for_mutual_accept, matched_pending_room, in_call
- is_online boolean default false
- last_seen_at timestamptz default now()
- current_session_id uuid nullable
- current_room_name text nullable
- updated_at timestamptz default now()

Note:
This table is app state. Supabase Realtime Presence can still be used in memory/channel form, but the DB table helps querying and auditing.

3) candidate_actions
Purpose:
Store accept/reject/block/report decisions between users.

Columns:
- id bigserial primary key
- actor_user_id uuid not null references profiles(id)
- target_user_id uuid not null references profiles(id)
- action text not null
  allowed values: accept, reject, block, report
- reason text nullable
- created_at timestamptz default now()

Indexes:
- index on actor_user_id
- index on target_user_id
- composite index on (actor_user_id, target_user_id, action)

4) active_matches
Purpose:
Track current mutual match / call session.

Columns:
- id uuid primary key default gen_random_uuid()
- user_a_id uuid not null references profiles(id)
- user_b_id uuid not null references profiles(id)
- status text not null
  allowed values: proposed, mutual_match_found, room_creating, joining_room, in_progress, ending, ended, failed
- matched_at timestamptz default now()
- started_at timestamptz nullable
- ended_at timestamptz nullable
- end_reason text nullable
- room_name text nullable
- provider text nullable default 'livekit'
- created_by text nullable default 'system'

Important constraints:
- prevent same unordered pair from having multiple simultaneous active rows
- maintain server-side validation that neither user is already active elsewhere

5) match_reservations
Purpose:
Short-lived reservation lock so one user cannot be assigned twice at the same moment.

Columns:
- id uuid primary key default gen_random_uuid()
- user_id uuid not null references profiles(id)
- reserved_for_match_id uuid not null references active_matches(id)
- expires_at timestamptz not null
- created_at timestamptz default now()

6) reports
Purpose:
Safety reporting.

Columns:
- id uuid primary key default gen_random_uuid()
- reporter_user_id uuid not null references profiles(id)
- target_user_id uuid not null references profiles(id)
- active_match_id uuid nullable references active_matches(id)
- reason text not null
- details text nullable
- created_at timestamptz default now()
- reviewed boolean default false

7) blocks
Purpose:
Permanent or long-term user blocking.

Columns:
- id uuid primary key default gen_random_uuid()
- blocker_user_id uuid not null references profiles(id)
- blocked_user_id uuid not null references profiles(id)
- created_at timestamptz default now()

Unique:
- unique(blocker_user_id, blocked_user_id)

8) seen_candidates
Purpose:
Track who has already been shown to whom.

Columns:
- id uuid primary key default gen_random_uuid()
- viewer_user_id uuid not null references profiles(id)
- candidate_user_id uuid not null references profiles(id)
- last_shown_at timestamptz default now()
- shown_count int default 1

Unique:
- unique(viewer_user_id, candidate_user_id)

9. ROW LEVEL SECURITY STRATEGY
======================================================================

Use Supabase RLS carefully.

Profiles:
- users can read public profile fields of other users if needed for candidate cards
- users can only update their own profile

Presence:
- users can read presence of users eligible to be surfaced
- users can only update their own presence row directly if allowed
- sensitive transitions such as in_call should preferably be updated via server-side function

Candidate actions:
- users can insert their own actions
- users should not read everyone else's decisions broadly
- server-side functions may aggregate this securely

Active matches:
- users can read matches where they are user_a_id or user_b_id
- users cannot arbitrarily insert room data from client
- room creation should be server-side

Reports and blocks:
- users can create their own
- users can read only their own reports/blocks unless admin workflow exists

Important:
All authoritative matching, reservation, and room-creation logic must be server-side, not trusted to the client.

11. VIDEO LAYER DESIGN (RECOMMENDED: LIVEKIT)
======================================================================

Use LiveKit Cloud for 1-to-1 call rooms.

Recommended room policy:
- one room per active match
- room name format:
  match_{matchId}
- server generates a room token for each participant
- only the two matched participants may join that room

Flow:
1. Mutual match is found server-side.
2. Server creates active_matches row.
3. Server generates room name.
4. Server creates or prepares LiveKit access token(s).
5. Server sends token and room info to both users via private realtime event or protected API response.
6. Both users join the room.
7. Call starts.
8. When one user ends the call:
   - leave room
   - notify other side
   - mark session ended
9. Use webhook callbacks to confirm participant join/leave and final room end.

LiveKit webhooks should be used for:
- participant joined
- participant left
- room finished
- recording hooks later if ever added
- audit logging
- cleanup if client disconnects unexpectedly

Frontend with LiveKit:
- use LiveKit React components and hooks where convenient
- build custom UI around them if needed

Minimum in-call controls:
- mute/unmute microphone
- enable/disable camera
- switch camera if mobile/native later
- end call
- left-swipe end gesture
- connection state indicator

14. BACKEND / EDGE FUNCTION RESPONSIBILITIES
======================================================================

Create protected server-side functions or APIs for:

1. create_or_update_profile
- ensure profile exists after auth
- update profile completion state

2. enter_queue
- validate authenticated user
- ensure user is not already in call
- set presence to queueing
- clear stale reservation if any
- return first candidate or acknowledgment

3. leave_queue
- remove from queueing state
- clear browsing state if needed

4. get_next_candidate
- fetch best next candidate according to rules
- exclude blocked/rejected/already-active users
- mark candidate as shown in seen_candidates
- return candidate profile

5. submit_candidate_action
Input:
- target_user_id
- action = accept | reject | block | report
Behavior:
- store action
- if block, create block row and suppress future surfacing
- if report, create report row
- if accept, check mutual acceptance
- if mutual, attempt atomic reservation and create match
- return either next candidate or match_found payload

6. create_match_if_mutual
This should be internal server logic, not a public raw client endpoint.
Behavior:
- verify both users free
- create match row
- create reservations
- prepare room
- update presence
- notify both clients

7. get_room_join_payload
- verify requester belongs to match
- return room URL/token/name
- only valid for active match

8. end_call
Input:
- match_id
- end_reason
Behavior:
- verify requester belongs to match
- mark ended
- release reservations
- clear room state in presence
- notify other participant
- optionally requeue requester automatically

9. auto_cleanup_stale_matches
- scheduled cleanup for failed rooms, abandoned reservations, stale queue states

15. MATCHING ALGORITHM PSEUDOCODE
======================================================================

Pseudo logic for get_next_candidate(user_id):

1. verify user authenticated
2. verify user not blocked / not in active call
3. build candidate pool:
   - users currently queueing
   - exclude self
   - exclude blocked both directions
   - exclude users with active in_progress or pending match
   - exclude recent rejects
   - exclude recently shown candidates if cooldown still active
4. order by:
   - not seen recently first
   - random weight second
   - active freshness
5. return first candidate
6. write/update seen_candidates

Pseudo logic for accept(userA, userB):
1. store action A accepts B
2. check if B already accepted A and that accept is still valid
3. if not mutual:
   return { status: "accepted_waiting", nextCandidate: maybe_next }
4. if mutual:
   begin transaction
   re-check both free
   create active_match
   create reservations
   update presence for both
   commit
   generate room access
   publish match_found to A and B
   return { status: "match_found", matchId, roomPayload }

16. IMPORTANT CONCURRENCY AND DATA INTEGRITY RULES
======================================================================

The AI agent must pay special attention here.

Problem:
User B may be accepted by multiple users at nearly the same time.

Required behavior:
- only one active match can win
- the winner is whichever transaction reserves B first
- all losing attempts must fail gracefully and fetch a new candidate

Rules:
- use server-side transaction
- use unique reservation logic
- check active session before finalizing
- use short-lived reservation TTL
- clean stale reservations automatically

Never rely on frontend timing for correctness.

Do not use webhooks as a replacement for realtime signaling.
Do not allow one user to be in multiple active calls at once.

Build around the exact concept described in this brief.
